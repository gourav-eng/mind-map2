# Node types and clone system for mind-map canvas

This change introduces two distinct node types (Task and Concept) and a clone system where a node can appear in multiple locations while keeping title and content synchronized bidirectionally. A three-panel split view provides a dedicated interface for browsing cloned nodes and navigating to their canvas positions. The implementation adds ~230 lines to a ~3100-line single-file app, touching the data model, node creation, content editing, context menus, canvas rendering, and a new panel UI.

**Watch for:** Deleting a source node orphans all clones with dangling `cloneSourceId` references (confirmed). `duplicateNode` and `pasteNode` blindly copy `cloneSourceId`, silently enrolling copies into clone sync groups they weren't meant to join (confirmed). The Outline view still renders status/priority controls for Concept nodes (confirmed).

---

## High-level view

The node type system is a lightweight tagging approach: a `nodeType` field gates badge rendering and hides the status/priority chrome on the canvas for concept nodes. The distinction is cosmetic on the canvas but not enforced in the outline view.

The clone system is anchored on a `cloneSourceId` pointer. Editing title or content on any member of a clone family propagates the change to all siblings and the source in a single pass through `updateNode`.

The three-panel split view (clone list + location display + navigation) is conditionally rendered beside the canvas, uses the edges array to derive prev/next chain context, and re-uses the existing `focusedNodeId` highlight mechanism for navigation. Panel widths are fixed at 600px combined, which will squeeze the canvas to near-zero on narrow viewports.

The primary structural gap is lifecycle management: nothing in the delete or duplicate/paste paths accounts for clone relationships, which means the data model can drift into inconsistent states during normal use.

## Orphaned clone references on delete

`deleteNode` filters the node out and removes its edges, but does not touch any node whose `cloneSourceId` points to the deleted ID. After deleting a source node, its clones remain in the workspace with a dangling pointer. The clone badge still renders (since `cloneSourceId` is truthy), but sync propagation in `updateNode` will silently no-op because the target ID no longer exists. The clone panel stops listing that family since no node's ID matches the orphaned `cloneSourceId` — the clones become invisible orphans that still display a CLONE badge.

```javascript
// deleteNode today:
const filteredNodes = ws.nodes.filter(n => n.id !== id);
// No cleanup of nodes where n.cloneSourceId === id
```

Fix: either cascade delete (`filteredNodes.filter(n => n.cloneSourceId !== id)`) or null out the reference (`filteredNodes.map(n => n.cloneSourceId === id ? {...n, cloneSourceId: null} : n)`).

## Duplicate and paste propagate clone membership

`duplicateNode` deep-clones via `JSON.parse(JSON.stringify(target))`, which preserves every field including `cloneSourceId`. The result is a node that looks like an independent copy (title gets " (Copy)" suffix) but is still part of the clone sync group — the next time the source's title changes, the duplicate's title will be overwritten silently.

Similarly, `pasteNode` spreads `clipData.node` into the new node, preserving whatever `cloneSourceId` the source had. A copy-paste of a clone creates a new sync participant in a different position.

Both should explicitly set `cloneSourceId: null` on the resulting node.

## Concept nodes in the outline view

On the canvas, the diff correctly wraps status/priority badges and the property editor in `node.nodeType !== 'concept'` guards. The outline view's `renderOutlineCard` was not updated — it unconditionally renders clickable priority and status badges for every node. Cycling these fields on a concept node changes data that is invisible on the canvas, creating a split-brain UX between the two views.

## Clone panel: edge-based location display

The location panel derives prev/next from `edges.find(e => e.target === instance.id)` and `edges.find(e => e.source === instance.id)`. Since `find` returns only the first match, nodes with multiple incoming or outgoing edges will show only one predecessor and one successor — heavily connected nodes will display an incomplete picture of their neighborhood.

## Panel layout and viewport interaction

The clone panels use fixed widths (`w-[250px]` and `w-[350px]`) with `shrink-0`, consuming 600px of horizontal space. On viewports below ~1000px the canvas becomes unusably narrow. There's no responsive breakpoint or mobile adaptation — on small screens the panels simply squeeze the canvas to near-zero width.
