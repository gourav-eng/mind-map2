# Hidden multi-project system with password-gated isolation (v2)

This change introduces a secret multi-project system where each project has isolated workspaces, its own SHA-256-hashed password, and independent state. Projects are managed through a hidden panel triggered via Ctrl+Shift+P or triple-tap on the app logo. The change also extends clone propagation across workspaces and adds a "Clone to Tab" context menu. Seven issues from the prior review have been addressed: passwords are now hashed, the `createProject` Date.now() quirk is resolved, the project panel is extracted to a shared render function, localStorage writes are debounced, the passwordless-project password-change flow skips the current-password check, and the dead reference-equality guard in `updateNode` is removed.

**Watch for:** Deleting the active project auto-switches to the next project in the array and sets `isAuthenticated(true)` without requiring that project's password — granting unverified access (confirmed). The debounce timer is scheduled inside React state updater functions, which is an unconventional side-effect placement that works today but is fragile under React Strict Mode double-invocation (likely).

## High-level view

All seven v1 issues are properly resolved. The SHA-256 hashing is applied at every entry point and a migration pass on initialization re-hashes legacy plaintext passwords. The shared `renderProjectPanel(isGate)` keeps both password-gate and full-panel rendering in a single code path.

The debounce implementation uses a shared `saveTimerRef` with 500ms trailing delay, placed inside the `setProjects` state updater. This prevents per-keystroke serialization but relies on React state updaters not being called multiple times (Strict Mode in development will double-invoke them, scheduling two timers). Functional in production builds, fragile under dev tooling.

The one new behavioral gap is in `deleteProject`: when the deleted project is also the active project, the function loads the first remaining project and sets `isAuthenticated(true)` unconditionally. A user who knows project A's password can delete it and gain access to project B without ever authenticating against B.

## Delete-project grants unverified access to fallback project

When `deleteProject` removes the active project, it loads the first remaining project and unconditionally authenticates:

```javascript
if (targetId === activeProjectId) {
  const next = updated[0];
  // ...loads next project's workspaces...
  setStoredPassword(next.password || '');
  setPasswordEnabled(!!next.password);
  setIsAuthenticated(!!next.password);  // grants access without password check
  localStorage.setItem('nexus-active-project', next.id);
}
```

The `deleteProject` function itself requires the *deleted* project's password (`hashedInput !== target.password`), but never asks for the *fallback* project's password. A user who only knows project A's credentials can delete A and immediately view project B's workspace data.

The fix: after deleting the active project, set `setIsAuthenticated(false)` so the password gate activates for the new project. The user will need to authenticate against the new active project (or switch to one they know the password for via the gate's "Switch" option).

## Debounce scheduled inside state updater

```javascript
setProjects(prev => {
  const updated = prev.map(p => p.id === activeProjectId 
    ? { ...p, workspaces, activeTab, nextId }
    : p
  );
  if (saveTimerRef.current) clearTimeout(saveTimerRef.current);
  saveTimerRef.current = setTimeout(() => {
    localStorage.setItem('nexus-app-state', JSON.stringify(updated));
  }, 500);
  return updated;
});
```

State updater functions should be pure — React reserves the right to call them multiple times (Strict Mode does this in dev). Each invocation schedules a new timer, and only the last one's `clearTimeout` targets the prior timer. In Strict Mode, this results in two timers being active simultaneously (the first isn't cleared because the second invocation's `clearTimeout` targets the timer scheduled by the first invocation of the same updater). The net effect in dev mode is a double-write to localStorage — benign but indicates the pattern violates React's expectations.

A cleaner approach: have the effect call `setProjects` purely (return `updated`, no side effects), then schedule the debounced write in the effect body using a ref to hold the latest projects value.
