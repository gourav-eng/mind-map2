# Hidden multi-project system with password-gated isolation

This change introduces a secret multi-project system where each project has isolated workspaces, its own password, and independent state. Projects are managed exclusively through a hidden panel triggered via Ctrl+Shift+P or triple-tap on the app logo. The visible "Password Protection" sidebar section is removed; password management now lives entirely within the secret panel. The change also extends the existing clone system to propagate title/content edits across workspaces and adds a "Clone to Tab" context menu for cross-workspace node cloning.

**Watch for:** Plaintext password storage in localStorage exposes all project passwords to any code with storage access (confirmed). The `createProject` function calls `Date.now()` twice in quick succession, producing potentially different values for the workspace id and `activeTab` — though a post-hoc fixup line corrects the mismatch, the dead `activeTab` assignment is confusing (confirmed). The project panel inside the password-gate early return is a stripped-down duplicate of the full panel, only offering "Switch" — creating two separate UI trees to maintain (confirmed). The auto-save effect calls `setProjects` inside itself (writing to localStorage inside a state setter), which could trigger cascading re-renders on every workspace keystroke (likely).

## High-level view

The project system turns the existing single-workspace-set application into a multi-tenant local app. Each "project" bundles workspaces, an active tab pointer, an id counter, and a password into a single JSON array stored under `nexus-app-state`. A migration path reads old `premium-*` keys, wraps them in a default project, persists to the new format, and cleans up old keys. The approach puts all project data — including all passwords — in a single localStorage value readable by any script on the domain.

The secret panel duplicates most of its JSX inside the password-gate return branch. The gate version is limited to "Switch" (so a locked user can switch to a different project), while the main app version has all four operations. Every panel layout fix needs to be applied in two places.

Clone propagation was previously workspace-local. The new `updateNode` lifts this to the full `workspaces` array, iterating all workspaces to find related clones. A new "Clone to Tab" context-menu action and the clone panel now surface clones across all workspaces, with navigation that switches tabs.

The password comparison is a strict-equality check of plaintext strings. No hashing, no timing-safe comparison, no rate limiting. The "password" is a UI gate, not a security boundary.

1. **Plaintext passwords in localStorage** — All project passwords are stored as raw strings in the `nexus-app-state` JSON value. Anyone (or any extension) with access to localStorage can read them. Consider hashing with `SHA-256` before storage if any meaningful confidentiality is intended.

3. **Duplicate panel JSX in password gate** — The project panel inside the `passwordEnabled && !isAuthenticated` early return is a manually-maintained copy of the main panel. Extract a shared component or render function to avoid drift.
4. **Auto-save triggers `setProjects` on every workspace change** — The `useEffect` watching `[workspaces, activeTab, nextId, initialized, activeProjectId]` calls `setProjects(prev => ...)` with `localStorage.setItem` inside. Every character typed in a node title triggers a full serialize-and-persist of the entire projects array. Debounce the localStorage write.
5. **No password-change validation for passwordless default project** — The "Change Key" flow requires entry of the current password, but the default migrated project may have `password: ''`. The user must submit an empty field to pass — a UX dead-end with no guidance.
6. **Export only serializes active project workspaces** — `exportData` still exports `{ workspaces, activeTab, nextId }`, losing all other projects on export/import.
7. **Dead reference-equality check in `updateNode`** — After `ws.nodes.map(...)`, the `nodesChanged` check compares by reference. Since `.map()` always returns a new array, this guard is always `true` and never short-circuits. Remove it or use a real equality check.

## Plaintext passwords and the localStorage trust model

Every project's password is stored as a raw string inside the `nexus-app-state` JSON array. The password comparison in `switchProject` and `deleteProject` is `projectPasswordInput !== target.password` — no hashing between user input and stored value.

Any browser extension, injected script, or DevTools access can read every project password:
```javascript
JSON.parse(localStorage.getItem('nexus-app-state')).map(p => ({name: p.name, password: p.password}))
```

The `nexus-active-project` key additionally reveals which project is currently loaded. A minimal improvement: hash passwords with `crypto.subtle.digest('SHA-256', ...)` before storage so the stored value doesn't directly reveal the credential.

## `createProject` and the `Date.now()` timing quirk

```javascript
const newProj = {
  id: `proj-${Date.now()}`,
  ...
  workspaces: [{ id: `ws-${Date.now()}`, ... }],
  activeTab: `ws-${Date.now()}`,   // third call — potentially different value
  nextId: 10
};
newProj.activeTab = newProj.workspaces[0].id; // fixup
```

Three separate `Date.now()` calls could produce different values on fast machines. The fixup line corrects `activeTab`, but the dead assignment is misleading. Assigning the workspace id to a local variable eliminates the ambiguity and the need for the fixup.

## Dual rendering of the project panel

The password-gate early return contains 60+ lines of duplicated panel markup supporting only "Switch" mode. If the gate's switch rendering drifts from the main panel's switch mode (different placeholder text, error handling, or styling), users will see inconsistent behavior depending on lock state. Extracting the switch sub-panel into a shared render function eliminates this risk.

## Auto-save performance under rapid edits

```javascript
useEffect(() => {
  if (initialized && activeProjectId) {
    setProjects(prev => {
      const updated = prev.map(p => p.id === activeProjectId
        ? { ...p, workspaces, activeTab, nextId }
        : p
      );
      localStorage.setItem('nexus-app-state', JSON.stringify(updated));
      return updated;
    });
  }
}, [workspaces, activeTab, nextId, initialized, activeProjectId]);
```

Every keystroke that updates a node title triggers this effect, which serializes the *entire* projects array to localStorage. For multiple projects with many nodes, `JSON.stringify` on large data blocks the main thread. A second `useEffect` watching `storedPassword` does the same serialization — a password change triggers two full serializations back-to-back.

Debouncing the localStorage write (500ms trailing) while keeping the in-memory state update immediate would preserve responsiveness.

## Dead reference check in cross-workspace `updateNode`

The `nodesChanged` variable after `.map()`:
```javascript
const updatedNodes = ws.nodes.map(n => { ... });
const nodesChanged = updatedNodes !== ws.nodes;
if (!nodesChanged && !isActiveWs && !hasEditedNode && !hasRelatedClone) return ws;
```

`.map()` always returns a new array reference, so `nodesChanged` is always `true`. This guard never fires. The effective short-circuit is the earlier check before the `.map()` — the second one is dead code that adds confusion.

## Export/import doesn't account for multi-project

`exportData` still serializes only `{ workspaces, activeTab, nextId }` — the active project's workspace data. Users who export and re-import will silently lose all non-active projects.

## Default project password UX gap

When migration creates a default project with `password: ''`, the "Change Key" flow checks `projectPasswordInput !== current.password`. The user must submit with an empty "Current key" field to pass — no visible guidance indicates this is possible or expected. Consider skipping the current-password check when the project has no password set.
