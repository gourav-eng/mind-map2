# Hidden multi-project system with password-gated isolation (v3)

This change implements a secret multi-project system with SHA-256-hashed passwords, isolated workspaces per project, and a hidden management panel (Ctrl+Shift+P / triple-tap logo). It also extends clone propagation across workspaces and adds a "Clone to Tab" context menu. Both issues from the v2 review have been addressed.

**Watch for:** Nothing blocking. The two v2 concerns (delete-project auth bypass and side effects inside state updaters) are both resolved.

**Verdict**: APPROVED

## High-level view

The delete-project auth bypass is fixed: `setIsAuthenticated(false)` now forces the password gate after deletion removes the active project, so the user must authenticate against the fallback project before accessing its data.

The debounce logic is properly extracted outside the state updater. The `setProjects` call is now pure, and the debounced `localStorage.setItem` fires in the effect body using `projectsRef.current` to read the latest value. A cleanup effect clears the timer on unmount.

## Delete-project auth bypass — resolved

The fix at line ~1213:

```javascript
setIsAuthenticated(false);
```

Runs only when `targetId === activeProjectId`, forcing re-authentication against the fallback project.

## Debounce outside state updater — resolved

The fix moves the debounce logic to the effect body, after `setProjects` returns:

```javascript
setProjects(prev => {
  const updated = prev.map(p => p.id === activeProjectId 
    ? { ...p, workspaces, activeTab, nextId }
    : p
  );
  return updated;  // pure — no side effects
});
// Debounced localStorage write (outside state updater)
if (saveTimerRef.current) clearTimeout(saveTimerRef.current);
saveTimerRef.current = setTimeout(() => {
  const currentProjects = projectsRef.current;
  localStorage.setItem('nexus-app-state', JSON.stringify(currentProjects));
}, 500);
```

The same pattern is applied to the password-save effect. `projectsRef` is synced via a dedicated effect, ensuring the debounced write always serializes the latest state.

Note: `createProject`, `switchProject`, and `changeProjectPassword` still call `localStorage.setItem` inside their state updaters. These are single-fire user-initiated actions (not continuous updates), so they don't create the timer-race fragility that the auto-save debounce had — functionally safe for one-shot operations.
