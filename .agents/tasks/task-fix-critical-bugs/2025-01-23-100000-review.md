# Breaking the realtime sync re-render loop

This change fixes React Error #310 (infinite re-render) caused by a circular dependency between the Supabase realtime subscription, workspace state setters, and the cloud-save effect. The core fix replaces a fragile timestamp-based self-echo guard with a structural approach: a `skipNextSaveRef` flag set before state updates from remote sources, a `lastSyncedRef` to short-circuit redundant `setProjects` calls, and a ref (`activeProjectIdRef`) that decouples the subscription effect from project-switch state changes. The change also auto-switches to newly created projects and tightens the triple-click logo timing.

**Watch for:** The `skipNextSaveRef` is a single boolean guarding a multi-setter sequence where any interleaved render could consume the flag before all setters fire (likely). The `lastSyncedRef` uses reference equality on the `workspaces` array, which is fragile against future code that constructs new arrays with identical content (confirmed).

## High-level view

The infinite loop arose because incoming realtime data called `setWorkspaces`, which triggered the cloud-save effect, which wrote back to Supabase, which echoed back through the subscription. The new approach sets `skipNextSaveRef = true` before any state update originating from the realtime handler, so the cloud-save effect skips the next trigger. A secondary guard (`lastSyncedRef`) prevents the projects-sync effect from re-running when workspaces/activeTab/nextId haven't actually changed by reference.

Removing `activeProjectId` from the subscription effect's dependency array prevents the channel from being torn down and recreated every time the user switches projects. The ref-based approach (`activeProjectIdRef`) gives the subscription handler access to the current project ID without coupling its lifecycle to it. The ref is synced via a dedicated effect, which introduces a latent ordering dependency under concurrent rendering.

The project-creation flow now performs a full project switch inline rather than just showing the project list, duplicating about 12 lines from `switchProject`.

1. **skipNextSaveRef race with batched setters** â€” `skipNextSaveRef.current = true` is set once before multiple state setters (`setWorkspaces`, `setActiveTab`, `setNextId`, `setActiveProjectId`). If React batches these into one render, the flag is consumed once and works. If any setter triggers an intermediate render (e.g., in a concurrent-mode transition or a future React version), the flag is consumed on the first render and the subsequent setters still trigger the cloud-save effect. Consider resetting the flag inside the save effect only after all expected dependencies have been observed, or use a counter instead of a boolean.

2. **lastSyncedRef reference equality assumption** â€” `lastSyncedRef.current.workspaces === workspaces` relies on the workspaces array being the exact same object reference. Any code path that creates a new array with identical content (e.g., a `map` or `filter` that returns the same items) would bypass the guard and re-trigger `setProjects`. Currently safe given the code structure, but fragile against future refactors.

3. **Duplicated project-switch logic in createProject** â€” The auto-switch block in `createProject` replicates 12+ lines from `switchProject` (localStorage writes, transform reset, history clear, auth state). Any future change to the switch sequence must be applied in both places. Extract a shared helper.

4. **activeProjectIdRef sync relies on effect ordering** â€” The ref is updated via a separate `useEffect(() => { activeProjectIdRef.current = activeProjectId }, [activeProjectId])`. If the realtime handler fires between the state update and the effect commit (possible during concurrent rendering), the ref could be stale. In practice React 18's batching makes this unlikely for synchronous code paths, but it's a latent ordering dependency.

## skipNextSaveRef as a single-shot guard

The cloud-save effect checks `skipNextSaveRef.current` and resets it to `false` on the same render:

```jsx
if (skipNextSaveRef.current) {
  skipNextSaveRef.current = false;
  return;
}
```

This works when React batches all the preceding setters (`setWorkspaces`, `setActiveTab`, `setNextId`) into a single render. React 18 does batch these in event handlers and effects. But the realtime subscription callback runs asynchronously from a Supabase WebSocket message â€” its batching behavior depends on whether React wraps it in `unstable_batchedUpdates`. If any setter commits independently, the flag is consumed on that partial render, and the next render (carrying the remaining state changes) hits the save effect unguarded.

The `createProject` call originates from a click handler, where React 18 reliably batches, so that instance is safer. The realtime handler instance is the one at risk. A counter (`skipSaveCountRef`) that decrements instead of a boolean that flips would be more robust â€” or wrapping the realtime setters in `ReactDOM.flushSync` / `unstable_batchedUpdates` to guarantee atomicity.
