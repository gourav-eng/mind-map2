# Client-side password gate for mind map app

A full-screen password overlay blocks the app on load when protection is enabled. The sidebar gains a "Password Protection" section with a toggle and password-change input. State lives in localStorage under `nexus-password-*` keys. The implementation is ~140 lines, stays within the existing single-file architecture, and follows the established localStorage persistence pattern gated by an `initialized` flag.

**Watch for:** Password stored in plaintext in localStorage (confirmed) — trivially bypassable by anyone with DevTools access. No brute-force mitigation on the login gate (confirmed). Empty-password edge case allows locking the app with no valid password set (likely).

## High-level view

The password gate renders as an early return inside the component, after all hooks and the outline-view JSX assignment. This avoids hooks-order violations and fully replaces the rendered UI rather than overlaying it. However, the full component logic (workspace parsing, node computations, all event handlers) still executes on every render while the gate is displayed — wasted work that could be eliminated by wrapping the gate at a higher level.

The password is stored as a raw string in localStorage — no hashing, no obfuscation. For a client-side-only personal tool the threat model is casual-deterrence only, but this should be stated explicitly or upgraded to at least a basic hash.

A gap exists in the init path: if localStorage contains `nexus-password-enabled: "true"` with an empty `nexus-password` value, the gate renders and an empty-string submission passes the equality check, effectively locking and then unlocking the app with an empty password.

<details>
<summary>Issues (3)</summary>

1. **Plaintext password in localStorage** — the password is stored as a cleartext string readable from DevTools or any XSS vector. Consider hashing before storage if the threat model extends beyond casual deterrence.
2. **No brute-force throttling** — unlimited rapid guesses are possible on the gate. A failed-attempt counter with exponential backoff or lockout would mitigate drive-by attempts.
3. **Empty-password edge case** — if localStorage contains `nexus-password-enabled: true` with an empty `nexus-password` value, the gate renders but an empty submission authenticates (empty string === empty string). Add a minimum-length guard on save and a fallback that disables protection when no valid password exists.

<summary>Details</summary>

## Plaintext credential storage

The password persisted via `localStorage.setItem('nexus-password', storedPassword)` is a raw string visible in DevTools and readable by any script running in the same origin. The comparison on the gate is a plain equality check (`passwordInput === storedPassword`). The security here is strictly cosmetic — protection against accidental access, not a motivated attacker.

Applying even an unsalted SHA-256 before storage would prevent casual reading of the password from the Application tab without meaningfully complicating the implementation. Alternatively, document the threat model explicitly so future contributors don't assume stronger guarantees.

## No rate-limiting on the password gate

The form submission handler has no throttle, counter, or lockout. A script can submit hundreds of guesses per second. Adding a `failedAttempts` counter that disables the submit button for a few seconds after 3–5 failures would be trivial and would match the app's UX polish.

## Empty-password state and toggle edge case

The sidebar save handler trims the input and only proceeds if truthy — that guards against empty saves from the UI. The gap is in initialization: the init effect sets `passwordEnabled` and `storedPassword` with two independent conditionals:

```jsx
if (savedPasswordEnabled === 'true') {
  setPasswordEnabled(true);
}
if (savedPassword) {
  setStoredPassword(savedPassword);
}
```

If `nexus-password` is absent or empty while `nexus-password-enabled` is `"true"`, protection activates with an empty stored password. The gate then authenticates on any empty submission. Collapsing to a single conditional fixes this:

```jsx
if (savedPasswordEnabled === 'true' && savedPassword) {
  setPasswordEnabled(true);
  setStoredPassword(savedPassword);
}
```

## Wasted computation behind the gate

The early return sits after ~1960 lines of hook calls, state derivations, and JSX variable assignments. All of that executes on every render even when the gate blocks access. For a personal tool this is negligible in practice, but extracting the gate into a thin wrapper component (`<PasswordGateWrapper><WorkflowApp /></PasswordGateWrapper>`) would make the boundary explicit and skip all inner computation until authenticated.
