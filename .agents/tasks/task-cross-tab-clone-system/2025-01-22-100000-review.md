# Cross-workspace clone system

The change extends the existing same-workspace clone mechanism to operate across all workspaces (tabs). Source nodes and clone instances are now discoverable globally in the Clone Panel, title/content sync propagates to clones in any workspace, a "Clone to Tab..." context menu option lets users place clones in other workspaces, and clicking a clone location in a different tab navigates there and focuses the node. The Clone Locations sub-panel also now shows all incoming/outgoing edges per instance (using `filter` instead of `find`).

**Watch for:** Double `setWorkspaces` call in `updateNode` where the second updater may overwrite the first within the same React batch (confirmed). Cross-workspace navigation relies on a 100ms `setTimeout` to wait for tab switch to complete — this is a race against React's render cycle with no guarantee (likely). Clone placement always lands at fixed coordinates (200, 200) regardless of viewport or existing nodes in the target workspace (confirmed).

## High-level view

Clone propagation moved from a single-workspace map operation to a two-phase update: first `updateActiveWorkspace` updates the edited node locally, then a separate `setWorkspaces` call propagates title/content changes globally. Both calls target the same state atom in the same synchronous function, creating a batching hazard where the global propagation may read stale state from the first call — or worse, overwrite it entirely.

Cross-workspace navigation uses `setActiveTab` followed by a hardcoded 100ms delay before measuring `workspaceRef` and computing the viewport transform. The timeout is a heuristic — if the tab switch triggers expensive layout (many nodes, images), the ref may not yet reflect the target workspace's DOM.

The Clone Panel now iterates all workspaces to build its source list and instance list, attaching workspace metadata (`_workspaceId`, `_workspaceName`, `_edges`) as ad-hoc underscore-prefixed properties on cloned node objects. This establishes a pattern of runtime object augmentation that grows harder to reason about as more cross-workspace features land.

<summary>Issues (4)</summary>

1. **Double setState race in updateNode** — `updateActiveWorkspace` and the subsequent `setWorkspaces` both target the same state atom. In React 18 batching, the second updater's `prev` may not include the first call's changes. Merge both into a single `setWorkspaces` call with the propagation logic applied in one pass.
2. **setTimeout-based navigation is fragile** — The 100ms delay after `setActiveTab` has no guarantee the new workspace has rendered. Replace with a `useEffect` that watches `activeTab` and a pending-focus target, triggering viewport adjustment after the workspace DOM is ready.
3. **Fixed clone placement at (200, 200)** — Cross-workspace clones always appear at the same absolute position regardless of the target workspace's current viewport or node density. At minimum, offset successive clones or derive position from the target workspace's viewport center.
4. **Ad-hoc object augmentation pattern** — Attaching `_workspaceId`, `_workspaceName`, `_edges` to cloned node objects creates implicit coupling between the panel renderer and this augmentation step. Consider a separate lookup structure or a dedicated `CloneInstance` type.

## Double setState in updateNode

The original `updateNode` was a single `updateActiveWorkspace` call that did both the direct edit and clone propagation in one atomic state transition. The refactored version splits this into two separate state updates:

```jsx
updateActiveWorkspace(ws => { /* update edited node */ });

// Then separately:
setWorkspaces(prev => { /* propagate to all workspaces */ });
```

`updateActiveWorkspace` itself calls `setWorkspaces`:

```jsx
const updateActiveWorkspace = useCallback((updater) => {
  setWorkspaces(prev => prev.map(ws => ws.id === activeTab ? { ...ws, ...updater(ws) } : ws));
}, [activeTab]);
```

In React 18, both calls are batched into a single render. The second `setWorkspaces` receives `prev` as the state *before* the batch — meaning its search for the edited node (`ws.nodes.find(n => n.id === id)`) will find the *old* version, not the one with `updates` applied. The code compensates by doing `{ ...found, ...updates }` when constructing `editedNode`, and by applying `{ ...n, ...updates }` to the node with matching `id` in the global pass. The end-state is likely correct in practice — but only because the same updates are re-applied redundantly in both passes. This is fragile: if a future change introduces derived fields computed from `updates` in the first pass, the second pass won't see them.

The fix is a single `setWorkspaces` call that handles both the direct update and cross-workspace propagation in one reducer.

## Cross-workspace navigation timing

When clicking a clone instance in another workspace:

```jsx
setActiveTab(instance._workspaceId);
setTimeout(() => {
  if (workspaceRef.current) {
    const rect = workspaceRef.current.getBoundingClientRect();
    // compute and set transform...
  }
  setFocusedNodeId(instance.id);
  setTimeout(() => setFocusedNodeId(null), 3000);
}, 100);
```

The transform calculation uses `instance.x` and `instance.y`, coordinates captured at panel-render time. If the target workspace hasn't fully reconciled its node positions (e.g., after a layout recomputation triggered by the tab switch), the viewport may center on stale coordinates. A `useEffect`-driven approach that reads coordinates from the newly-active workspace's state after render would be more reliable.

## Clone to Tab placement

`cloneNodeToWorkspace` places every cross-workspace clone at `(200, 200)`:

```jsx
const clone = {
  id: nextId.toString(),
  x: 200,
  y: 200,
  ...
};
```

If a user clones multiple nodes to the same target tab, they all stack at the same position. The same-workspace `cloneNode` offsets by `(+60, +60)` from the source position. Cross-workspace clones could derive a position from the target workspace's existing node bounding box or apply incremental offsets.

## Clone Panel: duplicate React keys on multi-edges

The refactored edge display maps over `prevNodes` and `nextNodes` arrays with `pn.id` / `nn.id` as keys. If a node has multiple edges to the same target (structurally possible with the current edge model), React would see duplicate keys, causing rendering bugs or silent drops. Using the edge ID or array index as part of the key would be safer.
